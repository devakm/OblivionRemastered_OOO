# OOORS Full Release Tracker — v2 Architecture (design proposal)

> Status: **proposal, awaiting review**. Nothing in this doc is implemented yet.
> Once approved, the implementation order at the bottom guides the rollout.

## Why we're rewriting

The v1 architecture (currently committed to `main`) treats this repo as a passive snapshot store: each release is ingested from `X:\games\Oscuro\Oscuro's_..._alphaNN\`, the working tree is wiped, refilled, committed, tagged. That works for backfilling existing history but **nullifies most of git's value going forward**:

| v1 limitation | Why it's a problem |
|---|---|
| `release/` is wiped on every ingest | Edits made in the repo get blown away — there's no in-place editing surface |
| Source folder is the canonical state | Repo is downstream/derivative, not the source of truth |
| Branches and PRs on text content are pointless | Next ingest from the source folder overrides anything you committed |
| One commit = one full release ingest | No atomic small changes ("fix one INI line") |
| No per-binary history without `git log` archaeology | Hard to answer "when did this `.pak` last change?" without spelunking |
| Per-release diff doc only knows file-level adds/removes/changes | Can't say *what* changed inside the main ESP — just that it changed |

v2 inverts the model: **the repo IS the source of truth for text content and metadata**, you edit in place, and "a release" is just a tag at a point you choose. Binary content lives outside git via GitHub Releases, but every binary has a per-file co-tracking record that turns the `.md` history into the binary's own version log.

## The model

### Three principles

1. **Edit in place.** The working tree is what you commit. No more wipe-and-refill.
2. **Track binaries by reference, not by value.** Each binary file has a sibling `.md` "co-file" that records which release(s) shipped which SHA, with direct download links. The binary itself never enters git.
3. **Make ESP changes diffable.** Each ESP gets a sibling `.records/` directory containing one JSON file per record type the ESP defines (ARMO.json, WEAP.json, CELL.json, …). The binary ESP never enters git, but the structured inventory does — and `git diff alpha90..alpha91 -- '*.records/**'` becomes the auto-changelog, narrowed to whichever record types actually changed.

### File anatomy at a glance

For a single binary content file, three artifacts live next to each other in the working tree:

```text
…/Oscuro's_Oblivion_Overhaul.esp           ← gitignored, present at staging time, packaged into .7z
…/Oscuro's_Oblivion_Overhaul.esp.md        ← committed: per-release tracking table
…/Oscuro's_Oblivion_Overhaul.esp.records/  ← committed directory (ESPs only): per-record-type JSON inventory
```

Non-ESP binaries get just the `.md`:

```
…/ObsidianItems.pak                        ← gitignored
…/ObsidianItems.pak.md                     ← committed
```

Pure text files have no co-files:

```
…/MagicLoader/Oscuro's_Oblivion_Overhaul_BOOK.json   ← committed directly
```

## What gets committed vs not

| Pattern | Committed? | Notes |
|---|---|---|
| `*.json`, `*.ini`, `*.lua`, `*.md`, `*.txt`, `*.html`, `*.css`, `*.py`, `*.cfg`, `*.xml`, `*.csv`, `*.yaml`, `*.yml`, `*.toml` | yes | Text source files |
| `*.esp.records/**/*.json`, `*.esm.records/**/*.json` | yes | ESP inventory bundle (per-record-type JSON files in a sibling directory) |
| `*.<binary-ext>.md` | yes | Per-binary release tracking co-files |
| `*.esp`, `*.esm` | **no** | Replaced by `.records/` directory for diffing + `.md` for tracking |
| `*.pak`, `*.ucas`, `*.utoc` | no | Large IoStore binaries |
| `*.dll`, `*.exe` | no | Native binaries |
| `*.dds`, `*.tga`, `*.png`, `*.jpg`, `*.jpeg`, `*.webp` | no | Image binaries |
| `*.uasset`, `*.uexp`, `*.umap`, `*.bsa` | no | UE asset blobs |

Distinct from v1: **all ESPs are gitignored regardless of size.** Earlier proposal had a 1 MiB threshold; with the `.records/` directory carrying the diffable structure, the binary ESP itself adds no incremental value to the repo — even small ESPs would just be opaque blobs git can't diff. The inventory is what we actually want.

## Co-file format (`*.<ext>.md`)

One `.md` per binary. Hand-authored prose at the top (optional, written once when the binary first appears). Below that: a **release table appended one row per release**, newest first.

Example, `Oscuro's_Oblivion_Overhaul.esp.md` after a few releases:

```markdown
# Oscuro's_Oblivion_Overhaul.esp

The main OOO TES4 plugin — leveled lists, NPCs, creatures, items, quests,
factions, dialog. Depends on Oblivion.esm.

| Release | SHA-256 | Size | Changed | Description | Download |
|---------|---------|------|---------|-------------|----------|
| alpha91 | `abc123def456` | 16.0 MiB (16,835,508 B) |   | [release alpha91](https://github.com/devakm/OblivionRemastered_OOO/releases/tag/alpha91) | [alpha91.7z](https://github.com/devakm/OblivionRemastered_OOO/releases/download/alpha91/alpha91.7z) |
| alpha90 | `abc123def456` | 16.0 MiB (16,835,508 B) |   | [release alpha90](https://github.com/devakm/OblivionRemastered_OOO/releases/tag/alpha90) | [alpha90.7z](https://github.com/devakm/OblivionRemastered_OOO/releases/download/alpha90/alpha90.7z) |
| alpha89 | `def789aaa111` | 16.0 MiB (16,800,000 B) | ★ | [release alpha89](https://github.com/devakm/OblivionRemastered_OOO/releases/tag/alpha89) | [alpha89.7z](https://github.com/devakm/OblivionRemastered_OOO/releases/download/alpha89/alpha89.7z) |
| alpha88 | `def789aaa111` | 16.0 MiB (16,800,000 B) |   | [release alpha88](https://github.com/devakm/OblivionRemastered_OOO/releases/tag/alpha88) | [alpha88.7z](https://github.com/devakm/OblivionRemastered_OOO/releases/download/alpha88/alpha88.7z) |
```

Conventions:

- **One row per release.** No collapsing — lookup is one O(1) read by tag.
- **Newest first.** `git checkout <oldtag> -- <co-file>` rewinds the file so its row 1 is `<oldtag>` with no later rows visible.
- **`Changed` column** marks rows where the SHA differs from the prior (older) row. Visual cue without sacrificing the per-release row.
- **SHA shown short (first 12 chars).** Full value lives in `manifests/<tag>.json`.
- **Size in human + exact bytes.** Catches sub-MiB drifts.
- **Description column** links to the GitHub Release page (notes + asset list, browser-friendly).
- **Download column** links to the asset directly: `…/releases/download/<tag>/<tag>.7z` — works as a bare HTTP fetch, no scraping.
- **Co-files do not ship** in the install `.7z`. The package step excludes `*.md` in `release/` from the archive payload.

## ESP inventory format (`*.esp.records/`)

One inventory bundle per ESP/ESM, in a sibling directory. The directory contains one JSON file per record type the ESP defines, plus a `_meta.json` for header info.

### `_meta.json`

```json
{
  "esp": "Oscuro's_Oblivion_Overhaul.esp",
  "tes4": {
    "version": 1.0,
    "masters": ["Oblivion.esm"],
    "next_object_id": "0xDEADBEEF"
  },
  "record_types": ["ARMO", "WEAP", "LVLI", "CELL", "NPC_", "QUST", "..."]
}
```

### `<TYPE>.json` — one per record type

Each is an array of records following a per-type schema. Examples:

`ARMO.json`:

```json
[
  {
    "formid": "0098F4",
    "edid": "OOOMelusPetiliusCuirass",
    "full": "Melus Petilius's Cuirass",
    "modl": "Armor/Glass/Cuirass.nif"
  }
]
```

`WEAP.json`:

```json
[
  {
    "formid": "007AE7",
    "edid": "WeapHeavenFuryLongsword01",
    "full": "Heaven Fury",
    "modl": "Weapons/Iron/Longsword.nif"
  }
]
```

`LVLI.json` (leveled-item lists):

```json
[
  {
    "formid": "00ABCD12",
    "edid": "LL0NPCWeaponLongsword100at15",
    "entries": [
      {"level": 15, "count": 1, "ref_formid": "007AE7"},
      {"level": 20, "count": 1, "ref_formid": "007AE6"}
    ]
  }
]
```

`CELL.json`:

```json
[
  {
    "formid": "00012345",
    "edid": "OOOArondarInterior01",
    "is_interior": true,
    "refs": [
      {
        "formid": "01234567",
        "edid": null,
        "base_formid": "0098F4",
        "base_edid_resolved": "OOOMelusPetiliusCuirass",
        "position_changed": false
      }
    ]
  }
]
```

Each `<TYPE>.json` is sorted deterministically (by FormID) so `git diff` is clean across regenerations.

### Records captured per type

| Record type | Fields captured |
|---|---|
| All "base object" types (ARMO, WEAP, NPC_, CREA, ALCH, ACTI, AMMO, BOOK, CLOT, CONT, DOOR, ENCH, FACT, INGR, KEY, MISC, SCPT, SGST, SLGM, SPELL) | FormID, EDID, FULL, MODL |
| LVLI / LVLN / LVLC (leveled lists) | FormID, EDID, list of `{level, count, ref_formid}` entries |
| CELL | FormID, EDID, exterior/interior flag, list of refs (REFR/ACHR/ACRE) |
| REFR / ACHR / ACRE (inside CELLs) | FormID, EDID (rare), base FormID, position-changed flag (boolean — exact coords omitted to keep diff signal-to-noise high) |
| QUST | FormID, EDID, FULL, list of stage indices (depth varies by need; start with summary) |
| DIAL / INFO | FormID, EDID, FULL (DIAL); INFO records summarized as count+EDIDs (full conversation trees deferred) |

This is the MVP scope. Per-record deeper fields (e.g., armor stats, weapon damage) can be added incrementally as a per-type "extras" key without breaking the schema.

### Localization handling

Oblivion Remastered appears to use inline `FULL` strings (verified during ESP MagicLoader JSON exports — the JSONs have literal display names). If we hit any localized-string-table-reference variants, the parser needs to resolve them against the matching localization file. Flag during implementation; defer if not encountered.

### Cross-reference: SyncMap UE5 paths

The inventory is **pure** — one binary in, one inventory out, no external state. SyncMap UE5 paths are NOT embedded.

A **separate join step** (`scripts/join_syncmap.py` or as part of the diff renderer) reads the current SyncMap INI and augments the displayed inventory with the UE5 path for each FormID. So a generated changelog snippet might look like:

> **Added armor:** `OOOMelusPetiliusCuirass` (FormID `0098F4`) → "Melus Petilius's Cuirass" → `/Game/Forms/items/armor/NDArmorLightCuirass5.NDArmorLightCuirass5` *(default)* / `/Game/Forms/items/armor/DEAHeavyCuirassOrder6.DEAHeavyCuirassOrder6` *(Deluxe)*

The join keeps the inventory file simple and lets the report show both SyncMap variants by reading both INIs.

### Diff to changelog

A `scripts/diff_records.py` renders the JSON-vs-JSON diff into a human-readable Markdown section that gets appended to `docs/per-release/<tag>.md` when a release is cut. Structure:

```markdown
## ESP changes — Oscuro's_Oblivion_Overhaul.esp

### Added
- Armor: `OOOMelusPetiliusCuirass` (FormID 0098F4) — "Melus Petilius's Cuirass"
- Weapon: `WeapHeavenFuryLongsword01` (FormID 007AE7) — "Heaven Fury"

### Changed
- Leveled list: `LL0NPCWeaponLongsword100at15` — added entry at level 20 (FormID 007AE6)
- NPC: `OOOMelusPetilius` — `FULL` changed from "Petilius the Knight" to "Melus Petilius"

### Removed
- Quest: `OOOOldDeprecatedQuest`

### Cell changes
- `OOOArondarInterior01`: added 5 REFRs (3 STAT, 2 ACTI)
- `Wilderness/-3,-2`: removed 1 REFR (ACRE)
```

That's the gold the user wants for README/changelog generation.

## Parser reuse strategy

We don't build from scratch. Three sources, ranked by coverage:

### Primary: Wrye Bash (`X:\dev\wrye-bash\`, `dev` branch)

Wrye Bash is the canonical mod-management toolchain for Oblivion / Skyrim / Fallout, and its `dev` branch already ships **explicit Oblivion Remastered support** alongside its existing Oblivion record library. The parser covers every record type we want — the per-record-type field extraction problem is already solved.

| Path | What it provides |
|---|---|
| `Mopy/bash/mod_files.py` | High-level entry: `LoadFactory` + `ModFile` classes that open a plugin and yield records. The `bosh` package wraps file/save handling around this. |
| `Mopy/bash/brec/record_structs.py`, `common_records.py`, `common_subrecords.py`, `complex_subrecords.py`, `record_groups.py` | Generic record/subrecord scaffolding (`MelStruct`, `MelString`, `MelGroups` etc.) — the DSL every `MreXxx` is built on. |
| `Mopy/bash/game/oblivion/records.py` | **64 `MreXxx` record classes** for every Oblivion record type: `MreArmo`, `MreWeap`, `MreCrea`, `MreNpc`, `MreLvli`, `MreLvln`, `MreLvlc`, `MreCell` (with REFR/ACHR/ACRE child handling via `AMreCell`), `MreActi`, `MreAlch`, `MreAmmo`, `MreBook`, `MreClot`, `MreCont`, `MreDial`, `MreInfo`, `MreDoor`, `MreEnch`, `MreFact`, `MreIngr`, `MreKeym`, `MreLigh`, `MreMisc`, `MreScpt`, `MreSgst`, `MreSlgm`, `MreSpel`, `MreQust`, etc. — each declares its full subrecord layout. |
| `Mopy/bash/game/oblivionre/__init__.py` | Oblivion Remastered game info (inherits `AOblivionGameInfo`, reuses oblivion's records as-is). Confirms TES4 record layouts are unchanged in OBR. |
| `Mopy/bash/patcher/patchers/` | The bashed-patch implementation — reference for *which* fields matter per record type when deciding "did this record meaningfully change?" |

#### Licensing — important

Wrye Bash is **GPL v3**. Two implications for us:

1. **Vendoring or `import`-ing wrye-bash code in our project would relicense this repo as GPL v3.** Currently this repo has no LICENSE file, so the default (proprietary) applies. If GPL v3 is acceptable for OOORS Full's tooling, this becomes a non-issue and we just `pip install wrye-bash` (or vendor a subset under `vendor/wrye_bash/` with the original COPYING file preserved).
2. **If we want to keep a non-GPL license**, we treat wrye-bash as **documentation only** — read `MreXxx` classes to learn which subrecords each record type uses, then write our own parser that re-implements the same knowledge. The TES4 record format is factual and not copyrightable; the code that parses it is. This is more work but keeps the licensing clean.

Recommendation: pick one approach as a deliberate decision (open question #8 below) before we touch the parser code.

### Secondary: in-house sister-project parsers

Useful for cross-reference and as a "second opinion" against wrye-bash's interpretation:

| Source | What we get |
|---|---|
| `X:\dev\OblivionRemastered_MapClone\fix_ooo_positions.py` | Generic GRUP/record/subrecord traversal, all GroupType values, compression handling, CELL/WORLD descent, REFR/ACHR/ACRE iteration, MAST/HEDR/EDID/FULL/NAME/XCLC/XTEL/XCLW subrecord parsing |
| `X:\dev\OblivionRemastered_MapClone\build_cellmap.py`, `batch_identity_clone.py` | Cell-block iteration patterns, persistence flag handling |
| `X:\dev\OblivionRemastered_ItemClone\fix_ooo_obsidian_lvli.py` | Leveled-list parsing (LVLD/LVLF/LVLO/LLCT/COED), WEAP record fields |

These are presumably your code, no GPL constraint. They're a useful baseline if we go the "re-implement, don't import" route — they already prove out the binary-traversal layer in pure Python with no external deps.

### What we still need to write fresh either way

Regardless of which parser route we take, the following layers are project-specific and have to be built here:

- **Inventory writer.** Walks parsed records, emits the `<esp>.records/<TYPE>.json` files (plus `_meta.json`) documented above. Format-stable across runs (sorted keys, FormID-sorted record arrays) so `git diff` is meaningful.
- **Inventory diff engine.** Compares two `.records/` directories; emits a structured diff (added / changed / removed, grouped by record type).
- **Diff-to-Markdown renderer.** Turns the structured diff into the changelog Markdown that gets appended to `docs/per-release/<tag>.md`. Joins against the current SyncMap INI to enrich each entry with its UE5 path(s).
- **Co-file appender.** Given a binary path + tag + sha + size, append a row to its `.md`. Idempotent.
- **Glue code.** A small Python module that wraps wrye-bash's (or our re-implemented parser's) output into our inventory schema. ~100-200 lines.

## Release command

A single end-to-end command replaces the v1 ingest+package+publish chain:

```powershell
py -3 scripts/release.py alpha91 --latest [--dry-run] [--no-publish]
```

What it does, in order:

1. **Validate working tree.** No uncommitted unrelated changes; required binaries present.
2. **Regenerate SyncMap variants** by running `sync_syncmap.py` against `release/` (uses OOO_SyncGen as source-of-truth INI).
3. **Hash the working tree** (text + binaries + co-files) → write `manifests/alpha91.json`.
4. **Parse all ESPs** in the working tree → write/update `<esp>.records/` directory contents.
5. **Update `*.md` co-files** for every binary present: append a row for `alpha91` with SHA + size + GitHub URLs.
6. **Diff vs prior tag's manifest + ESP inventories** → write `docs/per-release/alpha91.md` (file-level diff + ESP-content diff sections).
7. **Stage and commit** all changes (release/, manifests/, docs/per-release/).
8. **Tag** `alpha91`.
9. **Build** `dist/alpha91.7z` from `release/`, EXCLUDING `*.md` co-files and any `*.records/` directories.
10. **Publish** (unless `--no-publish`):
    - `git push origin main`
    - `git push origin alpha91`
    - `gh release create alpha91 dist/alpha91.7z --title alpha91 --notes-file docs/per-release/alpha91.md --latest`
11. **Refresh Pages** by running `sync_pages.py`. (Pages commit/push remains manual — Step 11 just stages.)

`--dry-run` prints every action without doing it. `--no-publish` does steps 1-9 locally, leaves push + release for later.

For a re-release of an existing tag (re-running with `--force`), the command:

- Deletes the local tag, re-runs steps 3-8.
- Re-pushes the tag with `--force-with-lease`.
- Calls `gh release upload alpha91 dist/alpha91.7z --clobber` to replace the asset.

## Unpacker utility

When you need a historical binary back on disk (e.g., to test alpha75's main ESP):

```powershell
# fetch one specific binary for one specific tag
py -3 scripts/fetch_binary.py --tag alpha75 release/.../Oscuro's_Oblivion_Overhaul.esp

# fetch every gitignored binary for one tag (full historical reconstruction)
py -3 scripts/fetch_binary.py --tag alpha75 --all

# show what would be fetched without downloading
py -3 scripts/fetch_binary.py --tag alpha75 --all --list

# fetch and overwrite the working-tree binary in place
py -3 scripts/fetch_binary.py --tag alpha75 release/.../Foo.pak --into-place
```

Mechanics:

1. Read the sibling `.md` co-file. Find the row for the requested tag → pull Download URL + expected SHA.
2. Cache the `.7z` at `work/_release_cache/<tag>.7z` so multiple fetches per tag download once.
3. Use `7z e` to extract the single requested file from inside the archive into a destination of your choice (default: `work/historical/<tag>/<original-relative-path>`; `--into-place` overwrites the working-tree binary).
4. Verify SHA-256 matches what the `.md` claimed. Bail loudly on mismatch.

## Migration plan (v1 → v2)

**Recap of current state:** 91 commits + 89 tags + `variants/alpha32nex` branch, all ingested via v1 model. 88 GitHub Releases live (alpha90 deferred, will become Latest after fixup). dist/ has 89 archives. `release/` contains text from alpha90.

The migration is destructive of the current main-branch history but **leaves the 88 already-published GitHub Releases untouched**. Per resolved decision #5, no re-upload is needed: the `.md` co-files record per-binary SHAs (which are invariant across `.7z` re-packaging), so the SHA we hash during backfill automatically matches what `fetch_binary.py` will extract from the existing GitHub assets.

### Migration steps

1. **Implement v2 tooling** (parser, release.py, co-file generator, diff renderer, fetch_binary.py).
2. **Lock in the v2 design via a dry-run** on a single recent alpha (say, alpha89): build the working-tree state for alpha89 in a scratch location and verify the `.records/` directory contents and `.md` outputs look right.
3. **Hard reset main** to the bootstrap commit; delete all 89 tags; delete `variants/alpha32nex` branch.
4. **Author the new tooling commit** (parser + release.py + supporting scripts + this design doc + updated CLAUDE.md/README.md).
5. **Backfill**: walk the 89 alpha source folders chronologically (plus alpha32nex on a variant branch). For each:
   - Replicate the source folder's content into `release/` (text + binaries; binaries gitignored but physically present so they get packaged).
   - Parse ESPs → write `.records/` bundle (per-type JSONs + `_meta.json`).
   - Append a row to every binary's `.md` co-file for this tag (note: at this point in backfill, the GitHub Release might not exist yet — we use the canonical asset URL pattern; Release publishing happens at the end).
   - Diff vs the prior tag's manifest + inventories → write `docs/per-release/<tag>.md`.
   - Build `dist/<tag>.7z`. (The pre-built ones from v1's prebuild can be reused since we already have them on disk.)
   - Commit + tag.
6. **Variants**: branch off `alpha32`, ingest `alpha32nex` on `variants/alpha32nex`, return to `main`.
7. **Push everything**: `git push origin main`, `git push origin --tags`, `git push origin variants/alpha32nex`.
8. **Re-upload the 88 archives** with `gh release upload <tag> dist/<tag>.7z --clobber` (skipping alpha90 still). This guarantees the SHA in each `.md` co-file matches the actual on-GitHub bytes.
9. **Refresh Pages** via `sync_pages.py`, commit + push the devnull repo.
10. **Author alpha90 fixup** in `release/` directly (in-place editing is now the workflow). When ready: `py -3 scripts/release.py alpha90 --latest --force`.

End state: same 89 tags, same 88+ Releases, but the repo is now an editing surface, every binary has full per-release tracking via `.md`, every ESP has a content inventory, and every `docs/per-release/<tag>.md` includes auto-generated "what changed in the ESP" sections.

## What stays the same

- `manifests/<tag>.json` — same format, same purpose. Captures SHA-256 of every file (text + binary) per release.
- `dist/<tag>.7z` — same packaging format and gitignored-from-repo status. Built from `release/` minus the `.md` co-files and `*.records/` inventory directories.
- GitHub Releases as the binary distribution channel.
- `sync_pages.py` writing HTML into `X:\dev\devnull\docs\OOO\OBR\`.
- `restrict-path.py` hook deny-list (game install + source folders read-only).
- Per-release diff doc at `docs/per-release/<tag>.md` — but now incorporates the auto-generated ESP-content diff sections, not just file-level changes.

## What goes away

- The "ingest from `X:\games\Oscuro\..._alphaNN`" mental model for new releases. Source folders are retired going forward (kept on disk for backfill input + archive only).
- `scripts/ingest_release.py` and `scripts/backfill_history.py` get folded into `scripts/release.py` and `scripts/backfill_v2.py`.
- The wipe-and-refill of `release/`. Replaced by edit-in-place.
- The 1-MiB ESP threshold debate. All ESPs gitignored; their inventory JSON is the diff surface.

| 6 | `fetch_binary.py` | Unpacker utility |
| 7 | `backfill_v2.py` | One-shot historical rebuild from source folders |
| 8 | Migration: reset → backfill → push (tags + main + variant branch) | The actual cutover. No re-upload needed (decision #5). |
| 9 | Updated docs | CLAUDE.md, README.md, release-process.md updated to describe v2; this v2-design.md retired or moved to a "history of redesigns" doc |

Estimated total effort: 2–4 days of implementation. Step 1 is the largest unknown (parser corner cases). Steps 2–6 are mostly straightforward Python given step 1. Step 7 is mechanical. Step 8 is a few hours of running scripts + waiting for uploads. Step 9 is doc writing.

## Resolved decisions (was: open questions)

All eight settled in the design review.

1. **REFR placement diff granularity → binary flag.** Each REFR record stores a `position_changed` boolean (compared to the same FormID's prior value). Exact coordinates are NOT kept in the inventory JSON. Cell-level diffs report counts ("3 REFRs added, 2 moved").

2. **QUST depth → header + stage indices.** Per-quest entry: FormID, EDID, FULL, list of stage indices. Stage script bodies skipped. Add later if needed.

3. **DIAL/INFO depth → counts + EDIDs.** Per DIAL: FormID, EDID, FULL, count of INFO children + list of their EDIDs. Conversation tree structure skipped.

4. **Inventory layout → per-record-type, in a sibling directory.** Each ESP gets a `<esp>.records/` directory containing one JSON file per record type, plus a `_meta.json` for the TES4 header:

   ```text
   Oscuro's_Oblivion_Overhaul.esp                 ← gitignored binary
   Oscuro's_Oblivion_Overhaul.esp.md              ← release tracking table
   Oscuro's_Oblivion_Overhaul.esp.records/        ← committed inventory bundle
     _meta.json                                   ← esp name, TES4 header, masters
     ARMO.json
     WEAP.json
     CELL.json
     LVLI.json
     NPC_.json
     QUST.json
     ... (one file per record type the ESP defines)
   ```

   Each `<TYPE>.json` is just an array of record entries with the per-type schema. The `.records/` directory is regenerated on every release ingest. Diff machinery picks up adds/removes at the directory level (whole record type added/removed) plus content diffs at the file level (records added/changed/removed within a type).

   This makes diffs much more focused: changes to ARMO records show up as a `git diff` to a single 50-KB file instead of a multi-MB whole-ESP file.

5. **Re-upload all 88 archives during backfill → no.** The `.md` co-files record per-binary SHAs (the SHA of `Foo.esp` itself), not per-archive SHAs (the SHA of `alpha85.7z`). Since the binary inside the existing `.7z` on GitHub is byte-identical to what's in the source folder we're hashing from, the `.md` SHA we record during backfill automatically matches what `fetch_binary.py` will extract. The archive-level framing differs but nothing checks it. Skip the re-upload entirely.

6. **Variants beyond alpha32nex → one-off.** `release.py` does NOT need branch-awareness as a general feature. The variant-branch handling is encoded only in the backfill script (which knows about `alpha32nex`); going forward, a hypothetical new variant would be handled with a manual `git checkout -b variants/<name> <parent-tag>` followed by `release.py`.

7. **DLL inventory → `.md` only, no records analog.** Confirmed.

8. **Wrye Bash licensing → adopt option (a), GPL v3.** Use wrye-bash directly. Add `LICENSE` file (GPL v3 text) to the repo before parser work begins. Vendor wrye-bash code under `vendor/wrye_bash/` with its original COPYING preserved (or `pip install` if a maintained PyPI release exists for the dev branch). Parser-effort estimate drops to ~half a day for a thin adapter.

## TL;DR for the impatient

Repo becomes an in-place editing surface. Binaries are gitignored, but each has a sibling `.md` that records every release's SHA + GitHub download URL (one row per release, newest first). ESPs additionally get a sibling `.records/` directory containing one JSON file per record type (ARMO.json, WEAP.json, CELL.json, …) plus `_meta.json` — diffs of those JSONs become auto-generated changelog sections in `docs/per-release/<tag>.md`. A single `release.py` command does ingest+package+publish end-to-end. The 89 historical alphas get re-backfilled into the new layout from the source folders; the 88 already-published GitHub Releases stay valid as-is (no re-upload needed — `.md` records per-binary SHA which is invariant across packaging). Going forward, "make a release" is `git commit + git tag + release.py` — no separate alpha source folder needed.
